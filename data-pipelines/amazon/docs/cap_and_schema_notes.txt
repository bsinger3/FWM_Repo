Approval Batch Capping and Schema Notes
======================================

Context
-------
We analyzed the approval-batch image files generated from Amazon review images.
The current usable-image rule is:

- usable if has_person_REVIEWONLY = true

The approval-batch enriched files live in:

- data/step_3_image_annotation/machine_annotated_outputs/


Current image-selection concern
-------------------------------
Some product/reviewer combinations contribute many image rows.
We do not necessarily want to delete those rows from the database, but we may
want to limit how many of them appear in search results.

Important distinction:

- Store all image rows in the database
- Do NOT necessarily return all of them in user-facing search results


Product-reviewer concentration findings
---------------------------------------
We wrote a report here:

- docs/product_reviewer_image_concentration_report.md

Headline findings from the finished chunk files:

- 91,925 image rows analyzed
- 30,279 unique product-reviewer combinations under the original profile-only
  grouping approach
- median = 2 images per product-reviewer combination
- max = 102 images for a single product-reviewer combination

This showed that concentration is real, but the original profile-only grouping
was overstating the problem because many rows had no reviewer_profile_url.


Better reviewer identity fallback
---------------------------------
We found that reviewer_profile_url is missing on many usable rows, so grouping
only by product + reviewer_profile_url is too aggressive.

Recommended reviewer identity rule:

1. Use reviewer_profile_url-derived profile ID when present
2. Otherwise fall back to reviewer_name_raw + review_date + hash(user_comment)
3. Otherwise fall back to review_date + hash(user_comment)
4. Otherwise fall back to hash(user_comment)
5. Otherwise fall back to reviewer_name_raw + review_date
6. Otherwise fall back to reviewer_name_raw

This logic is implemented for reporting in:

- scripts/report_reviewer_cap_impact.py

And summarized in:

- docs/reviewer_cap_impact_report.md


Cap impact using the smarter fallback
-------------------------------------
Using only rows where has_person_REVIEWONLY = true:

- Total usable images: 57,127

With smart reviewer fallback:

- cap 2 keeps 53,555 usable images and removes 3,572 (6.3%)
- cap 3 keeps 56,528 usable images and removes 599 (1.0%)
- cap 4 keeps 56,857 usable images and removes 270 (0.5%)

Important comparison:

- Earlier profile-only cap 2 estimate kept only 33,430 usable images
- Smart fallback cap 2 keeps 53,555 usable images

So the smarter fallback preserves 20,125 more usable images than the
profile-only approach.


How capped images should be handled
-----------------------------------
Decision so far:

- Images that exceed the cap should still be stored in the database
- But they should NOT be returned in search results by default

This means the cap should affect search eligibility, not physical storage.

Recommended model:

- keep every image row
- add status/eligibility metadata
- default search queries to only return eligible rows
- allow the capped-out rows to remain available for later reprocessing,
  auditing, or rule changes


Suggested database fields for future schema work
------------------------------------------------
When schema tracking is added to the repository, consider introducing fields
like these:

- search_eligible
- exclusion_reason
- cap_group_key
- cap_rank_within_group
- cap_priority_score
- reviewer_identity_method

Example meanings:

- search_eligible:
  true if the image should appear in default search results

- exclusion_reason:
  examples: over_reviewer_product_cap, low_quality, duplicate_candidate

- cap_group_key:
  a stable grouping key representing product + reviewer identity

- cap_rank_within_group:
  rank of the row after sorting within the product/reviewer group

- cap_priority_score:
  numeric score encoding the preference order, such as lighting and
  full-body visibility

- reviewer_identity_method:
  examples: profile, name_date_comment, date_comment, comment, name_date, name


Suggested future implementation behavior
----------------------------------------
When we implement the production cap logic:

1. Compute a reviewer identity using the smart fallback rule
2. Group rows by product_id + reviewer_identity
3. Rank rows within each group
4. Keep only the top N rows in that group as search_eligible = true
5. Mark the rest as search_eligible = false with exclusion_reason showing the
   cap outcome
6. Keep all rows stored in the database

Bottom-line recommendation
--------------------------
Best current direction:

- Do store all images
- Do cap search eligibility per reviewer/product group
- Do NOT rely only on reviewer_profile_url for reviewer identity
- Do use the smart fallback reviewer key
- A cap of 3 looks like a strong starting point if we want a lighter-touch cap
- A cap of 2 is possible, but it is more aggressive
