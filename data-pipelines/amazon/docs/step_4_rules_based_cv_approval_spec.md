# Step 4 Rules-Based CV Approval Spec

This document defines an implementation-ready, rules-based computer-vision
workflow for automatically classifying Step 4 image rows into exactly one of:

- `APPROVE`
- `REJECT`
- `REVIEW`

The workflow is intended for fresh Step 4 sheets that do not yet contain
computer-vision outputs.

This spec is deterministic and auditable. It does not use an LLM for approval
decisions or explanation generation.


## Goal

Approve images that are clearly usable, reject images that clearly fail the
visual requirements, and send borderline or conflicting cases to manual review.

The decision logic should reflect the real failure modes observed in labeled
data:

- no person visible
- multiple people visible
- person too small / too far away
- only part of the body visible
- framing is cropped or otherwise insufficient

Face presence is a supporting signal, not the primary decision driver.


## Required Models

### 1. Person Detector: YOLO Detect

Purpose:

- count people
- identify the main subject
- measure subject size in frame

Required output columns:

- `person_count_yolo_detect`
- `main_person_height_pct_yolo_detect`
- `main_person_bbox_area_pct_yolo_detect`


### 2. Pose Detector: YOLO Pose

Purpose:

- estimate how much of the body is visible
- identify severe partial-body crops

Required output columns:

- `body_coverage_score_yolo_pose`

Optional diagnostic columns:

- `person_count_yolo_pose`
- `main_person_height_pct_yolo_pose`
- `main_person_bbox_area_pct_yolo_pose`


### 3. Face Detector: YuNet

Purpose:

- provide a supporting face-presence signal
- help identify cleaner person-centered images

Required output column:

- `has_face_yunet`


### 4. Optional Diagnostic Face Detector: SCRFD

Purpose:

- disagreement checking only
- offline comparison only in version 1

Optional output column:

- `has_face_scrfd`


## Required Columns On Fresh Sheets

The CV pipeline must add these columns before the rules engine runs:

- `person_count_yolo_detect`
- `main_person_height_pct_yolo_detect`
- `main_person_bbox_area_pct_yolo_detect`
- `body_coverage_score_yolo_pose`
- `has_face_yunet`

Optional diagnostic columns:

- `has_face_scrfd`
- `person_count_yolo_pose`
- `main_person_height_pct_yolo_pose`
- `main_person_bbox_area_pct_yolo_pose`

Rules engine output columns:

- `cv_decision`
- `cv_reason_code`
- `cv_reason_summary`

Allowed values:

- `cv_decision` must be one of `APPROVE`, `REJECT`, `REVIEW`
- `cv_reason_code` must come from the fixed code list in this spec
- `cv_reason_summary` must be produced from a fixed lookup table, not free text


## Core Principles

1. Hard failures should be rejected.
2. Borderline cases should be reviewed, not force-approved or force-rejected.
3. Face detection should never be the sole reason for rejection.
4. Body coverage and subject size should dominate the decision.
5. The result must be reproducible from the columns alone.


## Threshold Constants

Version 1 threshold constants:

- `REJECT_NO_PERSON_COUNT = 0`
- `REJECT_MULTIPLE_PEOPLE_COUNT = 1`
- `REJECT_MIN_BODY_COVERAGE = 66.7`
- `REJECT_MIN_SUBJECT_HEIGHT_PCT = 0.50`
- `REJECT_MIN_SUBJECT_AREA_PCT = 0.12`
- `APPROVE_MIN_BODY_COVERAGE = 75.0`
- `APPROVE_MIN_SUBJECT_HEIGHT_PCT = 0.60`
- `APPROVE_MIN_SUBJECT_AREA_PCT = 0.15`

Interpretation:

- `body_coverage_score_yolo_pose < 66.7` means insufficient visible body
- `main_person_height_pct_yolo_detect < 0.50` means subject too small / too far
- `main_person_height_pct_yolo_detect >= 0.60` is a strong positive framing
- `main_person_bbox_area_pct_yolo_detect < 0.12` is a secondary weak reject cue
- `main_person_bbox_area_pct_yolo_detect >= 0.15` is a strong positive framing


## Decision Codes

Allowed `cv_reason_code` values:

- `NO_PERSON`
- `MULTIPLE_PEOPLE`
- `LOW_BODY_COVERAGE`
- `SUBJECT_TOO_SMALL`
- `CLEAR_PASS`
- `MISSING_CV_DATA`
- `BORDERLINE_BODY_COVERAGE`
- `BORDERLINE_SUBJECT_SIZE`
- `BORDERLINE_NO_FACE`
- `BORDERLINE_COMPOSITION`


## Reason Summary Lookup

`cv_reason_summary` must be generated by fixed code lookup:

- `NO_PERSON` -> `No person detected`
- `MULTIPLE_PEOPLE` -> `Multiple people detected`
- `LOW_BODY_COVERAGE` -> `Too little body visible`
- `SUBJECT_TOO_SMALL` -> `Person too small in frame`
- `CLEAR_PASS` -> `Single person, strong framing, enough body visible`
- `MISSING_CV_DATA` -> `Required CV data missing`
- `BORDERLINE_BODY_COVERAGE` -> `Body visibility is borderline`
- `BORDERLINE_SUBJECT_SIZE` -> `Person size is borderline`
- `BORDERLINE_NO_FACE` -> `Borderline framing and no face detected`
- `BORDERLINE_COMPOSITION` -> `Composition is borderline`


## Rule Order

Rules must be evaluated in this exact order.


## Step 1: Hard Reject Rules

### Rule R1: No Person Detected

Condition:

- `person_count_yolo_detect == 0`

Output:

- `cv_decision = REJECT`
- `cv_reason_code = NO_PERSON`
- `cv_reason_summary = No person detected`


### Rule R2: Multiple People Detected

Condition:

- `person_count_yolo_detect > 1`

Output:

- `cv_decision = REJECT`
- `cv_reason_code = MULTIPLE_PEOPLE`
- `cv_reason_summary = Multiple people detected`


### Rule R3: Body Coverage Too Low

Condition:

- `body_coverage_score_yolo_pose < 66.7`

Output:

- `cv_decision = REJECT`
- `cv_reason_code = LOW_BODY_COVERAGE`
- `cv_reason_summary = Too little body visible`


### Rule R4: Subject Too Small

Condition:

- `main_person_height_pct_yolo_detect < 0.50`

Output:

- `cv_decision = REJECT`
- `cv_reason_code = SUBJECT_TOO_SMALL`
- `cv_reason_summary = Person too small in frame`


## Step 2: Clear Approve Rule

### Rule A1: Clear Approve

Required conditions:

- `person_count_yolo_detect == 1`
- `body_coverage_score_yolo_pose >= 75.0`
- `main_person_height_pct_yolo_detect >= 0.60`
- `main_person_bbox_area_pct_yolo_detect >= 0.15`

Supporting note:

- `has_face_yunet = true` is a positive support signal, but it must not be
  required for approval in version 1

Output:

- `cv_decision = APPROVE`
- `cv_reason_code = CLEAR_PASS`
- `cv_reason_summary = Single person, strong framing, enough body visible`


## Step 3: Review Rules

If a row does not trigger a hard reject and does not qualify for clear approve,
it must be assigned to `REVIEW`.

Use the first matching review rule in the priority order below.
Implementation requirement:

- reject rules must be evaluated first
- approve rule must be evaluated second
- review rules must be evaluated third
- the first matching review rule wins
- the implementation must stop after the first matched terminal rule


### Rule V1: Borderline Body Coverage

Condition:

- `66.7 <= body_coverage_score_yolo_pose < 75.0`

Output:

- `cv_decision = REVIEW`
- `cv_reason_code = BORDERLINE_BODY_COVERAGE`
- `cv_reason_summary = Body visibility is borderline`


### Rule V2: Borderline Subject Size

Condition:

- `0.50 <= main_person_height_pct_yolo_detect < 0.60`

Output:

- `cv_decision = REVIEW`
- `cv_reason_code = BORDERLINE_SUBJECT_SIZE`
- `cv_reason_summary = Person size is borderline`


### Rule V3: Borderline No Face

Condition:

- `person_count_yolo_detect == 1`
- `body_coverage_score_yolo_pose >= 66.7`
- `main_person_height_pct_yolo_detect >= 0.50`
- `has_face_yunet == false`

Policy note:

- `has_face_yunet == false` must never be used as a standalone reject condition
- face absence may only contribute to `REVIEW`, never directly to `REJECT`

Output:

- `cv_decision = REVIEW`
- `cv_reason_code = BORDERLINE_NO_FACE`
- `cv_reason_summary = Borderline framing and no face detected`


### Rule V4: Borderline Composition

Condition:

- row did not match any reject rule
- row did not match approve rule
- row did not match earlier review rules

Output:

- `cv_decision = REVIEW`
- `cv_reason_code = BORDERLINE_COMPOSITION`
- `cv_reason_summary = Composition is borderline`


## Decision Pseudocode

```python
def classify_row(row):
    person_count = row["person_count_yolo_detect"]
    subject_height = row["main_person_height_pct_yolo_detect"]
    subject_area = row["main_person_bbox_area_pct_yolo_detect"]
    body_coverage = row["body_coverage_score_yolo_pose"]
    has_face = row["has_face_yunet"]

    if person_count == 0:
        return ("REJECT", "NO_PERSON", "No person detected")

    if person_count > 1:
        return ("REJECT", "MULTIPLE_PEOPLE", "Multiple people detected")

    if body_coverage < 66.7:
        return ("REJECT", "LOW_BODY_COVERAGE", "Too little body visible")

    if subject_height < 0.50:
        return ("REJECT", "SUBJECT_TOO_SMALL", "Person too small in frame")

    if (
        person_count == 1
        and body_coverage >= 75.0
        and subject_height >= 0.60
    ):
        return (
            "APPROVE",
            "CLEAR_PASS",
            "Single person, strong framing, enough body visible",
        )

    if 66.7 <= body_coverage < 75.0:
        return ("REVIEW", "BORDERLINE_BODY_COVERAGE", "Body visibility is borderline")

    if 0.50 <= subject_height < 0.60:
        return ("REVIEW", "BORDERLINE_SUBJECT_SIZE", "Person size is borderline")

    if person_count == 1 and body_coverage >= 66.7 and subject_height >= 0.50 and not has_face:
        return ("REVIEW", "BORDERLINE_NO_FACE", "Borderline framing and no face detected")

    return ("REVIEW", "BORDERLINE_COMPOSITION", "Composition is borderline")
```


## Column Type Expectations

Expected input formats:

- `person_count_yolo_detect`: integer-like numeric
- `main_person_height_pct_yolo_detect`: float between `0` and `1`
- `main_person_bbox_area_pct_yolo_detect`: float between `0` and `1`
- `body_coverage_score_yolo_pose`: float between `0` and `100`
- `has_face_yunet`: boolean-like string `true` or `false`

Normalization rules before evaluation:

- trim whitespace
- coerce numeric strings to floats/ints
- coerce `true` / `false` case-insensitively
- treat blanks as missing


## Missing Data Policy

If any required core column is missing or blank, the row must be routed to
review.

Required core columns:

- `person_count_yolo_detect`
- `main_person_height_pct_yolo_detect`
- `body_coverage_score_yolo_pose`
- `has_face_yunet`

Missing-data fallback:

- `cv_decision = REVIEW`
- `cv_reason_code = MISSING_CV_DATA`
- `cv_reason_summary = Required CV data missing`


## Output Column Order

For fresh sheets, append or insert the rules output columns after the required
CV columns:

1. `person_count_yolo_detect`
2. `main_person_height_pct_yolo_detect`
3. `main_person_bbox_area_pct_yolo_detect`
4. `body_coverage_score_yolo_pose`
5. `has_face_yunet`
6. `cv_decision`
7. `cv_reason_code`
8. `cv_reason_summary`

Optional diagnostics may follow after that.


## Normalization Requirements

Before rule evaluation:

- boolean-like cells must be normalized from strings such as `true` / `false`
- numeric cells must be parsed robustly even if stored as spreadsheet text
- blank strings, whitespace-only strings, and `None` must be treated as missing
- decision logic must run only on normalized values


## Batch Processing Workflow

This section defines how to run the rules engine across a large batch of Step 4
CSV files, produce a summary report for humans, isolate review rows into one
combined CSV, merge human edits back into the full batch, and prepare approved
rows for Step 5.


## Batch Inputs

The batch job operates on a collection of Step 4 CSV files.

Each input file must contain the standard Step 4 base columns. The CV pipeline
must enrich each input row with the required CV columns before the rules engine
runs.

Required batch inputs:

- one or more Step 4 CSV files
- each file must include a stable row identity

Recommended row identity fields:

- `source_file`
- `source_row_number`
- `original_url_display`

Implementation requirement:

- each row must receive a unique deterministic review key before the combined
  review CSV is generated

Recommended key:

- `review_row_key = "{source_file}::{source_row_number}"`


## Batch Output Columns

Every processed Step 4 CSV row must receive:

- `cv_decision`
- `cv_reason_code`
- `cv_reason_summary`

Rows routed to human review must also support:

- `human_decision`
- `human_reason_note`

Allowed `human_decision` values:

- blank
- `APPROVE`
- `REJECT`

`human_reason_note` is free text and optional.


## Batch Decision Procedure

For each Step 4 CSV file in the batch:

1. Add the required CV columns.
2. Normalize the CV values.
3. Apply the rules in the order defined above.
4. Write `cv_decision`, `cv_reason_code`, and `cv_reason_summary`.
5. Keep all rows in the processed source CSV.

At the end of the batch:

- all rows retain a CV-driven status
- only `REVIEW` rows are exported to the human-review CSV


## Human Review Export

All rows with `cv_decision = REVIEW` must be written into a single combined CSV.

Recommended output path pattern:

- `data/step_4_human_review_and_visibility_decisions/review_queue/step_4_combined_review_queue.csv`

Required columns in the combined review CSV:

- `review_row_key`
- `source_file`
- `source_row_number`
- `original_url_display`
- `product_page_url_display`
- `reviewer_profile_url`
- `reviewer_name_raw`
- `cv_decision`
- `cv_reason_code`
- `cv_reason_summary`
- `human_decision`
- `human_reason_note`

Recommended additional context columns:

- `user_comment`
- `height_raw`
- `weight_raw`
- `has_face_yunet`
- `person_count_yolo_detect`
- `main_person_height_pct_yolo_detect`
- `main_person_bbox_area_pct_yolo_detect`
- `body_coverage_score_yolo_pose`

Implementation requirement:

- the combined review CSV must include only rows with `cv_decision = REVIEW`
- `human_decision` must be blank on initial export


## Human Review Merge-Back

After the reviewer edits the combined review CSV, the batch procedure must merge
those human decisions back into the original processed Step 4 CSV files.

Merge key:

- `review_row_key`

Merge behavior:

1. Load the edited combined review CSV.
2. For each row with a nonblank `human_decision`, find the matching source row
   by `review_row_key`.
3. Write:
   - `human_decision`
   - `human_reason_note`
4. Write a resolved final decision column:
   - `final_decision`
   - `final_reason_code`
   - `final_reason_summary`

Final decision logic:

- if `cv_decision` is `APPROVE` or `REJECT`, copy it directly into
  `final_decision`
- if `cv_decision` is `REVIEW` and `human_decision` is populated, use the human
  decision as `final_decision`
- if `cv_decision` is `REVIEW` and `human_decision` is blank, leave
  `final_decision` blank or mark the batch as incomplete

Recommended final reason behavior:

- if `cv_decision` is not `REVIEW`, copy `cv_reason_code` and
  `cv_reason_summary`
- if `human_decision` overrides a review row, set:
  - `final_reason_code = HUMAN_OVERRIDE`
  - `final_reason_summary = Reviewed by human`


## Batch Summary Report

After the rules engine runs on the batch, write a markdown report for the human
to inspect.

Recommended output path pattern:

- `data/step_4_human_review_and_visibility_decisions/reports/step_4_cv_batch_report.md`

The report must include:

1. Total row counts

- total rows processed
- total rows `APPROVE`
- total rows `REJECT`
- total rows `REVIEW`

2. Decision histogram

- count of rows by `cv_decision`

3. Reject histogram

- count of rows by `cv_reason_code` for `REJECT`

4. Review histogram

- count of rows by `cv_reason_code` for `REVIEW`

5. Per-file breakdown

- each source file
- row count
- approve count
- reject count
- review count

Recommended markdown sections:

- `## Batch Totals`
- `## Decision Histogram`
- `## Reject Reasons`
- `## Review Reasons`
- `## Per-File Breakdown`

Recommended histogram format:

- markdown table with counts
- optionally add simple text bars for readability


## Post-Review Approval Export For Step 5

After human review has been merged back and all review rows are resolved, the
procedure must prepare approved rows for the Step 5 folder.

Input requirement:

- all rows in the batch must have a populated `final_decision`

Export rule:

- keep only rows where `final_decision = APPROVE`

Recommended output path pattern:

- `data/step_5_publish_ready_outputs/final_human_approved_batches/`

Recommended file behavior:

1. preserve the original batch file partitioning where practical
2. write one approved-output CSV per source file, or one combined approved CSV
   per batch job
3. remove temporary review-only columns if Step 5 should not receive them

Columns to exclude from Step 5 export unless explicitly needed:

- `cv_decision`
- `cv_reason_code`
- `cv_reason_summary`
- `human_decision`
- `human_reason_note`
- `final_reason_code`
- `final_reason_summary`
- `review_row_key`
- `source_file`
- `source_row_number`

Column to preserve if the downstream workflow still expects it:

- `Approved for publishing`

Recommended mapping:

- if `final_decision = APPROVE`, set `Approved for publishing = 1`


## End-To-End Batch Procedure

Implementation sequence:

1. Run CV enrichment on all Step 4 source CSV files.
2. Apply rules and write `cv_decision`, `cv_reason_code`,
   `cv_reason_summary`.
3. Write the batch summary markdown report.
4. Export all `REVIEW` rows into one combined human-review CSV.
5. Human edits the combined review CSV.
6. Merge human review results back into the processed batch.
7. Write `final_decision` and final reason fields.
8. Verify that no unresolved review rows remain.
9. Export all `final_decision = APPROVE` rows into the Step 5 output folder.

### Module 1: Batch Discovery

```python
def discover_step_4_csv_files(root_dir):
    files = []
    for path in all_files_under(root_dir):
        if path.suffix.lower() == ".csv" and path.name.startswith("images_to_approve_part_"):
            files.append(path)
    return sorted(files)
```


### Module 2: Add Row Identity

```python
def add_row_identity(rows, source_file):
    for row_number, row in enumerate(rows, start=2):
        row["source_file"] = source_file.name
        row["source_row_number"] = row_number
        row["review_row_key"] = f"{source_file.name}::{row_number}"
    return rows
```


### Module 3: CV Enrichment

```python
def enrich_row_with_cv(row, models):
    image_url = row["original_url_display"]
    image = fetch_image(image_url)

    yolo_detect = run_yolo_detect(image, models.yolo_detect)
    yolo_pose = run_yolo_pose(image, models.yolo_pose)
    yunet = run_yunet(image, models.yunet)

    row["person_count_yolo_detect"] = yolo_detect.person_count
    row["main_person_height_pct_yolo_detect"] = yolo_detect.main_person_height_pct
    row["main_person_bbox_area_pct_yolo_detect"] = yolo_detect.main_person_bbox_area_pct
    row["body_coverage_score_yolo_pose"] = yolo_pose.body_coverage_score
    row["has_face_yunet"] = yunet.has_face
    return row
```


### Module 4: Normalization

```python
def normalize_bool(value):
    if value is None:
        return None
    text = str(value).strip().lower()
    if text == "true":
        return True
    if text == "false":
        return False
    return None


def normalize_float(value):
    if value is None:
        return None
    text = str(value).strip()
    if not text:
        return None
    return float(text)
```


### Module 5: Rule Evaluation

```python
def evaluate_cv_rules(row):
    person_count = normalize_float(row["person_count_yolo_detect"])
    subject_height = normalize_float(row["main_person_height_pct_yolo_detect"])
    subject_area = normalize_float(row["main_person_bbox_area_pct_yolo_detect"])
    body_coverage = normalize_float(row["body_coverage_score_yolo_pose"])
    has_face = normalize_bool(row["has_face_yunet"])

    core_values = [person_count, subject_height, body_coverage, has_face]
    if any(value is None for value in core_values):
        return ("REVIEW", "MISSING_CV_DATA", "Required CV data missing")

    if person_count == 0:
        return ("REJECT", "NO_PERSON", "No person detected")

    if person_count > 1:
        return ("REJECT", "MULTIPLE_PEOPLE", "Multiple people detected")

    if body_coverage < 66.7:
        return ("REJECT", "LOW_BODY_COVERAGE", "Too little body visible")

    if subject_height < 0.50:
        return ("REJECT", "SUBJECT_TOO_SMALL", "Person too small in frame")

    if (
        person_count == 1
        and body_coverage >= 75.0
        and subject_height >= 0.60
        and subject_area is not None
        and subject_area >= 0.15
    ):
        return ("APPROVE", "CLEAR_PASS", "Single person, strong framing, enough body visible")

    if 66.7 <= body_coverage < 75.0:
        return ("REVIEW", "BORDERLINE_BODY_COVERAGE", "Body visibility is borderline")

    if 0.50 <= subject_height < 0.60:
        return ("REVIEW", "BORDERLINE_SUBJECT_SIZE", "Person size is borderline")

    if person_count == 1 and not has_face:
        return ("REVIEW", "BORDERLINE_NO_FACE", "Borderline framing and no face detected")

    return ("REVIEW", "BORDERLINE_COMPOSITION", "Composition is borderline")
```

### Module 8: Merge Human Decisions

```python
def merge_human_review(batch_rows, edited_review_rows):
    review_index = {row["review_row_key"]: row for row in edited_review_rows}

    for row in batch_rows:
        if row["cv_decision"] != "REVIEW":
            row["final_decision"] = row["cv_decision"]
            row["final_reason_code"] = row["cv_reason_code"]
            row["final_reason_summary"] = row["cv_reason_summary"]
            continue

        edited = review_index.get(row["review_row_key"])
        row["human_decision"] = edited.get("human_decision", "") if edited else ""
        row["human_reason_note"] = edited.get("human_reason_note", "") if edited else ""

        if row["human_decision"] in ("APPROVE", "REJECT"):
            row["final_decision"] = row["human_decision"]
            row["final_reason_code"] = "HUMAN_OVERRIDE"
            row["final_reason_summary"] = "Reviewed by human"
        else:
            row["final_decision"] = ""
            row["final_reason_code"] = ""
            row["final_reason_summary"] = ""

    return batch_rows
```

## Validation Checklist

Before release, validate all of the following:

1. Every input row gets a unique `review_row_key`.
2. Every input row receives a `cv_decision`.
3. Every `cv_decision` has a matching `cv_reason_code` and `cv_reason_summary`.
4. The review queue contains only `REVIEW` rows.
5. Every edited review row merges back to exactly one source row.
6. The final resolved batch contains no unresolved rows before Step 5 export.
7. The Step 5 export contains only `final_decision = APPROVE` rows.


## Release Recommendation

Recommended implementation order:

1. implement CV enrichment
2. implement rules evaluation
3. implement markdown batch reporting
4. implement combined review queue export
5. implement human merge-back
6. implement Step 5 export

Do not combine all of this into one monolithic script. Keep each phase
separate so that the outputs are inspectable and rerunnable.

## Version 1 Recommendation

Version 1 production recommendation:

- auto-approve only `APPROVE`
- auto-reject only `REJECT`
- route all `REVIEW` rows to manual review

This gives you a conservative hybrid workflow that reduces manual effort while
keeping borderline cases in human hands.


## Version 1 Non-Goals

Do not require these in version 1:

- `has_face_scrfd`
- `has_face_blazeface`
- MediaPipe pose visibility columns
- any LLM-generated summary or rationale
- any scoring column

Reason:

- they are weaker, unavailable, blank, or unnecessary for the first rules-based
  approval engine
